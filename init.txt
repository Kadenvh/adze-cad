Plugin Modalities and Performance Characteristics One of the most critical architectural decisions for an AI developer is the choice of plugin modality. There are three primary ways to interact with the SOLIDWORKS process: VBA Macros, In-Process Add-ins, and Out-of-Process Stand-alone applications.17 The selection of modality has profound implications for execution speed, especially for AI tasks that require high-frequency geometric querying. VBA macros are the simplest form of automation, integrated directly into the SOLIDWORKS user interface.19 They are ideal for rapid prototyping and simple task automation, but they are fundamentally unsuited for modern AI integration due to the limitations of the VBA language and the lack of support for modern neural network libraries.17 In-process add-ins are the professional standard for AI integration.17 Because they run within the same memory space as the SLDWORKS.exe process, calls to the API are "in-process" and avoid the overhead of the Windows marshaling system.20 This is vital for AI applications that must traverse thousands of geometric entities to build a feature representation. An in-process add-in can perform geometry indexing up to 800 times faster than an out-of-process counterpart.20 Out-of-process stand-alone applications are separate executables that connect to SOLIDWORKS via the Running Object Table (ROT).13 While they offer greater flexibility and can be written in any language, they suffer from a "marshaling penalty" that makes them significantly slower.20 For example, indexing 17,000 faces in a large assembly might take only 3.43 seconds for an in-process add-in, while the same operation could take over 50 minutes for a stand-alone application.20 Modality Execution Context Performance Best Use Case VBA Macro In-Process High Simple repetitive tasks, prototyping 17 Add-in (DLL) In-Process Highest Real-time AI, geometry traversal, native UI 17 Stand-alone (EXE) Out-of-Process Low Background processing, multi-session control 13 To mitigate the performance issues of stand-alone applications, developers can utilize a "hybrid" approach involving the OnIdle notification. In this model, the stand-alone application schedules a job in a queue maintained by a "thin" in-process add-in. When the SOLIDWORKS thread becomes idle, the add-in executes the high-speed geometric queries and returns the result to the stand-alone application via a callback mechanism.20 This strategy allows for the flexibility of a stand-alone app while maintaining nearly the performance of a native add-in. Geometric Data Extraction for AI Training and Inference The efficacy of an AI model in a CAD environment is entirely dependent on the quality and structure of the data extracted from the model. Developers must master the traversal of two distinct but related data structures: the FeatureManager Design Tree and the Boundary Representation (B-Rep) geometry.21 The FeatureManager Design Tree represents the procedural history of the model. AI applications such as command prediction or workflow automation rely on this sequential data.5 Traversal is typically achieved using the IFirstFeature and IGetNextFeature methods.21 Once a feature reference is obtained, the developer can call GetSpecificFeature2 to access specialized interfaces, such as IRefPlane or IExtrudeFeatureData2, which contain the parametric definitions of the feature.21 B-Rep geometry, however, is the "source of truth" for the physical shape. B-Rep models decompose a solid into a hierarchy of shells, faces, loops, edges, and vertices.22 For AI models like BRepNet, which are designed to operate directly on solid models, the primary unit of topology is the "coedge"—an oriented edge that defines the relationship between a face and its neighbors.24 The SOLIDWORKS API provides methods to extract this information, allowing the developer to perform "topological walks" to identify mating faces and adjacent edges.24 For AI tasks involving mesh-based neural networks or point cloud analysis, the plugin can use the Tessellation object to convert smooth B-Rep surfaces into a collection of facets.25 This object allows for the configuration of mesh quality, including the requirement for vertex normals and face-to-facet mappings, which are essential for feeding standard computer vision models.25 Specialized data can also be extracted for specific AI use cases. For example, a plugin focused on automated mesh generation for simulation would use the CWMesh interface within the SOLIDWORKS Simulation API to extract nodal and element data.26 For AI-driven reverse engineering, the Scanto3D API provides methods to extract points and facets from mesh features imported from 3D scans.6 Bridging the Inference Gap: C# to Python Integration A fundamental challenge in AI development for SOLIDWORKS is the architectural disconnect between the CAD ecosystem and the AI research community. While SOLIDWORKS plugins are optimally developed in C#, the vast majority of AI models and training pipelines are implemented in Python.11 Bridging this gap requires an efficient inference layer that minimizes latency and data serialization overhead. There are several established approaches to this integration, each with distinct trade-offs in terms of performance and complexity. The ONNX Universal Inference Strategy The Open Neural Network Exchange (ONNX) format serves as a universal bridge for AI models. A model trained in Python using PyTorch or TensorFlow can be exported to an .onnx file and then loaded directly into a C# add-in using the Microsoft.ML.OnnxRuntime library.11 This approach is highly recommended for production environments because it eliminates the need for a local Python installation on the user’s machine and offers significant performance advantages through optimized C# bindings.11 The gRPC Microservices Architecture For more complex AI systems where the model is too large to run locally or requires specialized hardware (e.g., a Large Language Model), a gRPC-based microservices architecture is the preferred solution.28 gRPC uses the Protocol Buffers (Protobuf) binary format to define service contracts between the C# client (the SOLIDWORKS add-in) and a Python server.28 This is significantly faster and more compressed than traditional text-based REST or SOAP APIs, making it ideal for the high-volume geometric data transfers common in CAD automation.28 Python.NET and Process-Based Interop For rapid prototyping or use cases where performance is less critical than flexibility, developers can use Python.NET to call Python code directly from C#.14 This allows for the execution of Python scripts within the managed.NET environment of the plugin. 5. The context schema the model should receive

Do not send raw pixels or full geometry by default. Send a compact, typed context pack.

Host/session
SOLIDWORKS version, document type, units, active configuration, read-only/dirty state, cloud/offline state, user policy level.

Selection pack
Entity type, owning feature/component, stable identifier, nearby dimensions, mates or references, and why the selection matters.

Model summary
Feature-tree slice, driving dimensions, suppression state, mates, custom properties, rebuild warnings, reference graph, and document relationships.

Policy pack
Allowed tools, approval thresholds, export restrictions, PDM mode, and redaction rules.

Trace pack
The current plan, last tool results, warnings, and rollback anchors.

A good rule is: summary first, neighborhood on demand. Fetch more geometry or topology only when a specific tool needs it.

6. The approval and safety model

Use four execution classes.

Green — read-only or visual actions such as inspect, explain, highlight.
Auto-run.

Yellow — reversible single-document edits such as a dimension change or feature suppression.
Preview required, one-click confirm.

Orange — multi-object or export actions such as batch export, drawing generation, or macro execution.
Explicit scope summary and confirm.

Red — destructive, cross-document, or vault-affecting actions.
Typed confirmation, allowlist, and possibly admin policy.

Every action should create a trace record with before-state summary, after-state summary, warnings, user approval state, and rollback guidance. Native undo is helpful when it works, but the system should not rely on undo alone; it should also store compensating action logic where possible.
