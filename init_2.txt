The report’s old fixed price/perpetual framing is stale. Current official pages publish annual online pricing for named-user and device-license offers, with Standard, Professional, and Premium currently listed at $2,820, $3,456, and $4,716 per year in the U.S. Product pages also now reference SOLIDWORKS Ultimate, so the public packaging story is broader than the older Standard/Professional/Premium-only framing. Treat old price points as obsolete for planning.

The Microsoft compatibility section also needs updating. The current support matrix lists SQL Server 2025 as active with SOLIDWORKS 2026, SQL Server 2019 supported through SOLIDWORKS 2026 SP5, and Office 2024 supported from SOLIDWORKS 2025 SP3.

2) Native SOLIDWORKS AI is more real now, but still not the product you want

The report understates how much AI SOLIDWORKS is already shipping. Official materials now show Command Predictor in 2025, AI fastener recognition in 2025 SP3, and 2026 marketing around AI-powered drawing automation plus AURA. SOLIDWORKS’s AI Companions page now says SOLIDWORKS Design includes beta access to AURA and LEO, and that users on SOLIDWORKS Design with Cloud Services should activate their 3DEXPERIENCE connection to use them. At the same time, the detailed help entry for Auto-Generate Drawings (BETA) is still under the 3DEXPERIENCE-connected help flow and explicitly says 3D EXPERIENCE users can auto-generate drawings. So the current state is: native AI exists, but it is still tied to Dassault’s cloud-connected layer and does not replace a grounded desktop action assistant for your workflow.

3) The report overstates tier gating for custom add-ins

I did not find official documentation saying the general custom add-in model is Professional-only. What the official docs do show is a distinction between SOLIDWORKS add-ins and Other add-ins, and the add-in creation docs describe how to create/register custom add-ins generically. I would treat licensing gates as applying to specific built-in modules, APIs, or solution features—not to the base existence of a third-party add-in.

4) Document Manager’s role should be narrower

The report is right that Document Manager matters, but it should not be positioned as the live control plane. Official docs describe it as an independent component that does not require SOLIDWORKS to be installed, requires a subscription-gated license key, and can only access files created in the same or earlier SOLIDWORKS version than the key supports. That makes it excellent for closed-file indexing, metadata extraction, dependency graphs, and batch retrieval, but not for the live interactive assistant loop in the open desktop session.

5) Anthropic fit is good, but the runtime boundary must be fixed up front

Anthropic’s current docs make the Messages API the clean default: it is the primary API, it supports tool use, and current docs also expose official client SDKs including C#. Anthropic’s C# SDK is official but still marked beta. The Claude Code SDK has been renamed to the Claude Agent SDK, and the Agent SDK is currently positioned around Python and TypeScript with the Claude Code-style agent loop and persistent environment/session model. Anthropic also says third-party developers generally may not offer claude.ai login for products built on the Agent SDK. That pushes your v1 architecture toward Messages API + your own client-side tool loop, not “drop the Agent SDK inside SOLIDWORKS and let it own the session.”

6) Security/governance is not designed strongly enough

For proprietary CAD, this part cannot be loose. Anthropic’s docs say the Messages API is ZDR-eligible, and data residency controls exist through inference_geo and workspace geography. But the same docs say code execution, programmatic tool calling, and computer use are not ZDR-eligible, and computer use is explicitly beta. That means your production path for sensitive engineering data should stay on client-side tools over the Messages API, with the actual CAD actions executed by your software—not by Anthropic-hosted code execution or desktop-control betas.

7) MCP is useful, but the report doesn’t bound it tightly enough

MCP is worth using, just not as the foundation of the product. Anthropic’s tool docs show that Messages API can consume MCP tools directly, and the MCP connector can connect directly to remote MCP servers. But the connector is currently beta, currently supports tool calls only, and Anthropic’s remote MCP docs explicitly say those servers are third-party services that are not owned or endorsed by Anthropic. Anthropic’s Agent SDK docs also say MCP servers can run as local processes, over HTTP, or in-process. So MCP is a great compatibility boundary, but your core executor should still live inside the SOLIDWORKS add-in.

8) Existing agent tooling is useful for benchmarking, not for handing over architecture ownership

There is enough current tooling to benchmark against. MecAgent is a real benchmark for user expectations: task automation, direct in-CAD copilot behavior, engineering Q&A, simple text-to-CAD, part finding, and enterprise options including on-prem/private cloud. xCAD is the most useful reusable developer framework I found: examples for Task Pane WPF hosting, PropertyManager pages, MSI installers, and even .NET 6 add-ins. Open SolidWorks MCP projects are real and worth mining, but they also expose the pain points: SolidPilot uses a Python layer plus a C# adapter and PythonNET COM bridge; one newer Node-based server advertises an “intelligent COM bridge” plus dynamic VBA fallback because direct Node COM calls break on higher-parameter SolidWorks methods. Those are reference architectures, not the core I would ship.

The hard decisions I would make now

Use a native add-in, not an overlay-first bot.
The right insertion point is a Task Pane sidebar for persistent conversation and status, plus PropertyManager pages whenever the assistant needs selections, previews, or guided confirmations. That is the path that actually “understands and uses the UI” without pretending pixels are the primary truth.

Keep all SOLIDWORKS execution in-process.
Even the open community performance work still points the same way: out-of-process control is much slower for heavy geometry traversal, while in-process add-ins remain the natural high-performance path. Use a local broker for model calls and retrieval, but keep the actual SOLIDWORKS executor in the add-in.

Use Anthropic Messages API first.
For v1, choose Messages API + structured outputs + client-side tools. It is the clearest fit for auditability, approval gates, enterprise control, and data-governed CAD actions. Agent SDK becomes interesting later, once your tool catalog, traces, and policies are already solid.

Make MCP optional plumbing, not the product.
Expose your internal tool catalog as MCP later if you want Claude Desktop, Agent SDK, or other MCP clients to use it. Do not make Claude Desktop plus a public MCP server your primary UX or execution path.

Reserve computer-use and UI automation for fallback only.
Anthropic’s own docs say computer use is beta, not ZDR-eligible, and requires a sandboxed environment with strong safety precautions. That makes it a good research lane for unsupported dialogs, not a sound foundation for your first serious release.

The staged roadmap
Phase 0 — Freeze the envelope

Pick one SOLIDWORKS year, one deployment mode, one security mode, one document scope, and one UI surface stack. My recommendation is: SOLIDWORKS 2025/2026 desktop, parts + assemblies + drawings, Task Pane + PropertyManager, Anthropic Messages API, no PDM writes, no UI automation in production.

Exit criteria: six ADRs signed: host runtime, broker runtime, model/runtime policy, data locality mode, installer/update path, supported document/version envelope.

Phase 1 — Add-in shell and read-only grounding

Build the add-in shell, registration/installer, Task Pane, settings, logging, and the read-only context tools.

The assistant must answer, reliably:

what document is open

what is selected

what the active configuration and units are

what the relevant feature-tree slice is

what driving dimensions/custom properties/mates exist

what rebuild warnings or missing references exist

Exit criteria: the assistant answers grounding questions correctly across a benchmark set of parts, assemblies, and drawings before any write tools ship.

Phase 2 — Safe write tools

Add a narrow write layer with plan → preview → apply → verify → log.

Start with:

select/highlight entities

set dimension value

suppress/unsuppress feature

rename feature/component

set custom property

export PDF/STEP/DXF

run approved macro

Exit criteria: high task completion on a curated task suite, zero unapproved writes, and every write has a human-readable action trace.

Phase 3 — Guided authoring

Now let it help create, not just inspect. Start with guided drawing workflows first, then basic modeling.

Drawing workflows are the best early authoring target because outcomes are reviewable and the product already has native AI overlap there. Use PropertyManager pages for every operation that needs face/edge/plane confirmation or a preview state.

Exit criteria: it can complete a narrow list of authoring flows reproducibly with preview and confirmation.

Phase 4 — Teach mode

Do not try to “train it” from raw pixel recordings as the core method. Record a structured trace instead:

user intent

live CAD context

tool sequence

before/after summaries

user approvals

corrections

final outcome

Then compile successful traces into reusable automations, macro wrappers, and few-shot exemplars. That is how the assistant actually gets better inside a governed CAD environment.

Phase 5 — Headless knowledge and retrieval

Add Document Manager indexing and file/PDM-aware retrieval here, not earlier.

Capabilities:

closed-file metadata extraction

reference graph indexing

similar design lookup

standards/manual retrieval

prior automation trace retrieval

cross-document impact hints

Exit criteria: the assistant can answer questions that go beyond the active file without opening everything in SOLIDWORKS.

Phase 6 — Advanced geometry intelligence

Only after the live action loop is stable should you invest in:

tessellation pipelines

topology/coedge extraction

mesh/simulation feature access

image/scan-to-mesh assistance

design-rule models

feature-recognition models

This is where B-Rep ML and richer geometry embeddings become worth the cost.

Phase 7 — Optional interoperability layer

Once the native product is solid:

expose the same internal tool catalog through a local MCP server

test an Agent SDK sidecar for higher-autonomy flows

build a sandboxed computer-use fallback for unsupported dialogs

But keep all three behind flags until the core is already good.

Phase 8 — Enterprise hardening

Add:

code signing

MSI/managed deployment

proxy support

offline/degraded mode behavior

tenant/workspace policy controls

telemetry and rate-cost controls

retention policies

audit exports

admin allowlists

xCAD is useful here because it already ships examples for Task Pane hosting, installers, and newer add-in targets.

The context schema the model should see

Keep it compact and typed.

Host/session
SOLIDWORKS version, document type, units, active configuration, dirty/read-only state, offline/cloud state, policy level.

Selection pack
entity type, owner feature/component, document/config scope, nearby dimensions, mate/reference neighborhood, user intent hint.

Model summary
feature-tree slice, suppression state, key dimensions, custom properties, mates, rebuild warnings, reference graph.

Policy pack
allowed tools, approval thresholds, export restrictions, PDM mode, data-classification flags.

Trace pack
current plan, last tool outputs, warnings, rollback anchors, unresolved ambiguities.

Rule of thumb: summary first, neighborhood on demand.

The approval model

Use four classes.

Green — inspect, explain, highlight.
Runs automatically.

Yellow — reversible single-document edits.
Preview + one-click confirm.

Orange — exports, batch actions, macro execution, drawing creation.
Explicit scope summary + confirm.

Red — destructive, cross-document, vault-affecting, or external-sharing actions.
Typed confirmation + admin allowlist.

PropertyManager-page preview support is already part of the official UI model, which is one reason it is such a good fit for confirmation-heavy actions.

The main risks and how I’d handle them

COM/threading brittleness
Keep all SOLIDWORKS calls inside one in-process executor; never let remote agents own COM directly.

Hallucinated actions
Strict schemas, allowlisted tools, preview-first writes, and entity-resolution checks.

Ambiguous selections
Use PropertyManager-guided selection whenever intent is not uniquely grounded.

UI freezes
Broker model calls out of process; only execute short CAD mutations inside the add-in.

Data leakage
Messages API + client-side tools for production; no server-side code execution for sensitive CAD unless policy explicitly allows it.

Overbuilding too early
Do not start with B-Rep ML, overlay-first control, or full autonomy.
